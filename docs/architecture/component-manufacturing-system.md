# 组件式制造系统讨论记录

> Status: discussion note — 先记录设计方向，后续实现前再拆任务。

## 目标

做一个比固定配方更自由的制造系统。玩家可以通过 UI 或自然语言表达想法，系统根据材料、机制组件、处理方式和技能生成动态物品。

第一阶段建议从做饭开始验证，因为做饭天然是“食材 + 处理方式 + 调味/药性”的组合系统。

## 核心原则

- 物品不应该由固定形状定义。
- 形状、外观、名称都是描述性信息，不作为机制验证依据。
- 机制效果来自可验证组件和属性。
- 系统负责规则判定，包括材料消耗、难度、技能、效果、失败结果。
- LLM 只负责命名和描述，不负责决定真实效果、消耗或难度。

## 不采用的方案

- 不让 LLM 直接生成任意 Lua 来决定物品效果。
- 不要求所有动态物品都注册到 `Items._ALL`。
- 不通过枚举 `物品 × verb × 组合` 写大量 reaction。
- 不试图验证“6 面骰子是否真的有 6 面”这类物理形状。

## 推荐模型

制造结果由以下部分组成：

- 材料
- 组件
- 处理方式
- 可选意图
- 系统计算出的效果
- LLM 生成的名称和说明

动态物品可以存入 `item_instances`，使用 `physics_props`、`base_effects`、`displayed_effects`、`tags` 等字段保存实例信息。

## 做饭系统示例

玩家说：

> 我想做一个草药肉汤，这样可以让那些闻不下药味的人把药吃下去。

系统解析事实：

- 食材：肉、水、草药
- 处理方式：炖或煮
- 意图：遮盖药味

系统计算：

- 饱食恢复
- 体力恢复
- 草药症状缓解
- 药味强度
- 适口性
- 失败风险

LLM 生成说明：

> 姜艾肉汤用肉汤的咸香和油脂压住草药苦味，适合闻不下药味的病人饮用。它能缓解恶心并恢复体力。

## 组合爆炸问题

不能写成：

- 草药 `bake` 一条 reaction
- 草药 + 肉 `bake` 一条 reaction
- 草药 + 肉 + 盐 `bake` 一条 reaction

这样会爆炸。

应该拆成三层规则：

- 通用处理规则
- 组件交互规则
- 少量特例覆盖

优先级：

```text
特例 > 交互规则 > 通用处理规则
```

## 做饭规则示例

草药有性质：

- 药效
- 苦味
- 水溶性
- 热稳定性
- 香气

肉有性质：

- 饱腹
- 油脂
- 鲜味
- 遮盖苦味能力

水有性质：

- 汤底
- 提取水溶性药效

炖煮处理有性质：

- 温和加热
- 保留较多药效
- 提升融合度

结果由这些性质组合计算，而不是查固定配方。

## 难度来源

难度不来自形状。

难度来自：

- 处理方式复杂度
- 食材数量
- 药性复杂度
- 目标意图复杂度
- 材料品质
- 技能匹配度

示例：

- 烤肉：低难度
- 肉汤：中低难度
- 咸肉汤：中等难度
- 草药肉汤：中高难度
- 多草药病号汤：高难度

## 材料消耗

- 主料成功时消耗。
- 调味料成功时消耗。
- 水按量消耗。
- 工具不消耗，只扣耐久。
- 失败时按失败等级决定材料返还或损坏。

## LLM 边界

LLM 输入是系统事实。

LLM 输出只允许名称和描述。

LLM 不能：

- 新增治疗效果
- 降低难度
- 修改材料消耗
- 绕过技能检定

## 第一阶段建议

先做动态料理制造系统。

支持：

- 食材槽
- 处理方式
- 意图文本
- 预览
- 动态食物输出

动态食物可被食用，效果走现有 `ItemEffects` 路径。

LLM 只生成 `display_name` 和 `description`。
