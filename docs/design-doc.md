# 设计文档

> LLM 驱动、玩家自然语言创造内容的开放世界小镇游戏。
> 本文档只覆盖**游戏设计**（玩法、机制、内容、世界规则）。技术架构、沙箱、验证记录、工具链见 [tech-doc.md](./tech-doc.md)。

---

## 1. 游戏总览

**一句话**：在一个 Smallville 规模的魔法小镇里，玩家通过自然语言创造物品和法术，由 LLM 生成沙箱化代码实时执行；NPC 也会观察、模仿、自行创造，整个小镇形成涌现式的物品/知识/经济生态。

**类型**：开放世界 + 社会模拟 + 玩家驱动 UGC + 沙箱魔法 RPG

**核心创新**：
- 自然语言 → LLM → 沙箱代码 → 世界中可用对象
- 内容主要由玩家和 NPC 创造，开发团队只提供"物理定律"
- 平衡通过 API 级硬编码代价系统达成，不依赖白名单约束玩家创意

**规模**：单镇 ~50 NPC + 20~50 在线玩家（参考 Smallville 论文 + 早期 EVE 单星系/早期 MUD 单服务器的甜蜜点）

---

## 2. 核心设计原则

1. **代价是平衡的根本**：任何强力效果都有相应代价，代价由 API 层硬编码强制执行，不由 LLM 或玩家声明
2. **创造自由优先于设计师约束**：API 表面尽可能广，但每个原语带成本
3. **知识具象化**：API 知识不是数字精通等级，而是世界里可拾取/教授/逆向的物
4. **NPC 是一等公民**：NPC 也消耗资源、也创造、也持有知识，跟玩家在同一套规则下生活
5. **小而独立优于大而互通**：每镇是独立服务器，各自演化出文化路径

---

## 3. 三层资源经济

```
┌─────────────────────────────────────────┐
│ 第三层：知识 / 配方（永久积累）           │
│   - 法术书、技术书、API 知识             │
│   - 来源：拾取、师承、逆向工程            │
├─────────────────────────────────────────┤
│ 第二层：Mana（瞬时能量，有上限，会回复）  │
│   - 释放/使用创造物时按 API 调用扣除     │
│   - 上限通过消耗稀有魔法资源永久增长      │
├─────────────────────────────────────────┤
│ 第一层：物理资源（库存，会消耗）          │
│   - 木、石、铁、铜、银、金、草药...      │
│   - 稀有：魔晶、灵髓（用于扩 mana 上限） │
│   - 固定刷新时间（节点式，类 Stardew）   │
│   - NPC 也会采集和消耗                   │
└─────────────────────────────────────────┘
```

**关键机制**：
- Mana 上限不通过 XP/等级提升，通过消耗稀有魔法资源永久 +mana
- 这取代了"杀怪升级"的经典 RPG 循环，把驱动力换成探索 + 采集 + 解谜
- 采集节点固定刷新，玩家与 NPC 共享、互相竞争

## 4. 知识系统（替代精通系统）

**核心**：API 知识是世界里的"物"，不是角色面板上的数字。

| 获取方式 | 描述 |
|---|---|
| **法术书 / 技术书** | 古遗迹、矿洞深处、商人、任务奖励、NPC 持有；读完永久学会某段 API |
| **NPC 师承** | 每个 NPC 天生掌握某些 API；通过帮工/送礼/任务建立关系后会教 |
| **逆向工程** | 拆解他人创造物，消耗时间+资源，学会其中用到的 API |

**关键规则**：
- **知识一旦学会，无法被夺走**（脑内的）
- **法术书/技术书可被偷、可被抢**（物理对象）
- 玩家和 NPC 都可以**写法术书**（把脑内知识落到物上，可分享/出售）
- 法术书可复印（消耗资源），原本与抄本无差别
- 这天然形成知识的扩散/秘传/盗窃/教育生态

## 5. 创造系统

### 物品和法术由玩家通过自然语言创造

```
玩家描述："一把会让人说真话的剑"
       ↓
LLM 生成沙箱化代码（基于已知 API）
       ↓
代码被验证、注册到玩家的物品库
       ↓
使用时实时执行，按 API 调用扣 mana
```

技术细节（沙箱语言、API 表面、生成流程）见 [tech-doc.md §1-2](./tech-doc.md)。

### 基础品 vs 进阶品
- **基础物品 / 基础法术**：官方预设，每个新镇默认存在（确保 day-1 可玩）
- **进阶物品 / 进阶法术**：玩家和 NPC 创造出来，构成内容主体

### NPC 也会创造
- NPC 会观察玩家的创造物
- 会模仿、改编、独立创造
- 创造频率/质量受其属性（智慧、专业、关系网）限制
- NPC 创造物会进入小镇市场流通

## 6. 战斗与死亡

**战斗存在**：
- 玩家可以杀玩家、杀 NPC（PvP/PvE 都允许）
- 外部威胁存在（强盗、野兽、远方危险区）
- 这给"造武器/造防具"以真实理由

**杀人代价**：
- Mana 高昂（API 内硬扣）
- Smallville 记忆系统：NPC 会记仇、报警、断绝交易
- 守卫 NPC 会响应
- 这套社会代价大部分是涌现的，不需要硬编码

**死亡惩罚**：掉一些东西（中等惩罚）
- 不是永久死亡
- 不是完全无惩罚
- 具体掉落规则待细化（比例？随机？）

## 7. NPC 系统（Smallville 基础）

**核心架构来自 Generative Agents 论文**：
- Memory stream：所有经历存为带时间戳的自然语言记忆
- Reflection：定期总结记忆，形成更高层认知
- Planning：基于记忆与目标制定行动计划

**额外特性**（适配本游戏）：
- NPC 也消耗物理资源（吃饭、烧火、买东西）
- NPC 也持有 API 知识（可教授玩家）
- NPC 也会创造物品（参见 §5）
- NPC 关系网会因玩家行为动态变化

**镇主题与 NPC**：每个镇的主题、NPC 名册、人格/职业/亲属/秘密由开发者手工预制。LLM 不参与建镇。

---

## 8. 小镇与服务器架构

**镇即服务器**：每个小镇是一个独立的服务器实例。

**关键决策**：
- ✅ 只有官方能开新镇（由策划/开发者手工预制）
- ✅ 玩家不能跨镇（角色被绑定在所属镇）
- ✅ 一个玩家可以在多镇各有一个独立角色（数据互不互通）
- ✅ 每镇有人口上限，满了开新镇

**镇的种子内容**（开发者手工预制，LLM 不参与）：
- 地形与建筑布局
- NPC 名册、人格、社会关系
- 起始法术书 / 技术书
- 起始物品配方
- 镇主题（如"沿海渔村，盛产珍珠，居民信奉海神"）

**资源初始化**：每次新开服时，按预制的"区域 + 类型 + 密度"约束随机生成节点坐标（例：`north_forest` 区域内放 10±3 个 `wood` 节点）。运行期按固定刷新规则补充。

**镇的离线行为**：慢 tick（NPC 1 天 = 现实 1 小时之类），玩家在线时加速。具体节奏待定。

**镇的生命周期**：
- 长期未解决问题：老镇内容积累过多，新人难以竞争
- 是否引入赛季/重置/迁徙机制 → 待定，第一年内必须想清楚

## 9. 视觉与美术方案

**核心抉择**：放弃"每个玩家创造的物品都有独特视觉"，改用**有限图标词汇表 + LLM 修饰符**。

**理由**：
- AI 生图工具（如 PixelLab）当前实测效果不可用
- 3D 模型生成（Meshy 等）慢、贵、风格难统一
- D&D / Diablo / Roguelike 几十年都证明：物品独特性靠**名字 + 描述 + 行为**，视觉词汇有限完全可接受
- 玩家从"会让人说真话的剑"获得的想象远强于任何 AI 生成的具体图像

**方案构成**：

| 内容 | 方案 |
|---|---|
| 环境 / 建筑 | **3D，Synty Fantasy Kingdom 等资产包** |
| NPC 角色 | **3D，Synty 自带 + 后续补充** |
| 物品视觉（背包/UI） | **2D 图标词汇表**（手工 ~30-50 基础类型） |
| 物品装备到角色 | 武器：少量 3D 模型 + 颜色/光效变化；服装暂不可见 |
| 物品视觉变化 | LLM 在创造时选：基础类型 + 颜色/材质 + 修饰符（发光/火焰/寒霜/滴血/星光...） |
| 法术 VFX | MVP：Godot 内置 GPU 粒子 + 简单 shader；后期：买 Hovl/JMO/Kenney VFX 包 |
| UI | 常规 2D |

**示例**：
```
玩家：会让人说真话的剑
LLM 选择：
  - base: sword
  - tint: silver_white
  - effect: pale_blue_glow
代码：truth_compulsion 行为脚本
```

**代办**：
- 评估 Synty Fantasy Kingdom 内 NPC 角色覆盖度，决定是否补充其他角色包
- 设计图标词汇表的具体基础类型清单
- 设计修饰符的具体清单与组合规则

## 10. 已决策事项快速索引

| 决策 | 结论 |
|---|---|
| 平衡机制 | API 级硬编码 mana 成本 |
| 进度模型 | Mana 上限 + 知识=物，不用经典 XP/等级 |
| 资源 | 三层（物理 / mana / 知识） |
| 资源刷新 | 固定时间节点 |
| NPC 行为 | 也消耗资源、也创造、也持有知识 |
| 战斗 | 存在，PvP+PvE+外部威胁 |
| 死亡惩罚 | 掉一些东西（中等） |
| 知识获取 | 法术书 / NPC 师承 / 逆向工程 |
| 知识可被偷 | 否（脑内不可夺）；但法术书可被偷 |
| 基础品 vs 进阶品 | 基础官方预设，进阶玩家/NPC 创造 |
| 镇架构 | 独立服务器，仅官方能建 |
| 跨镇 | 不能 |
| 多角色 | 一玩家可在多镇各有独立角色 |
| 镇规模 | 单镇 ~50 NPC + 20~50 玩家在线 |
| 视角与画风 | 3D 环境/NPC + 2D 图标物品 + 粒子 VFX |
| 物品视觉策略 | 有限图标词汇表 + LLM 修饰符，放弃"每物品独特视觉" |
| 建镇方式 | 开发者手工预制（地图 / 建筑 / NPC / 起始书）+ 资源按约束随机初始化；LLM 不参与建镇 |

技术栈相关决策（沙箱语言、引擎、资产包、网络模型等）见 [tech-doc.md §5](./tech-doc.md)。

## 11. 待决策事项（按优先级）

### 高优先级（影响核心玩法）
1. **基础 API 表面设计**：哪些学派、每个学派的原语清单、mana 成本公式 → 下一轮专题讨论
2. **基础物品 / 基础法术清单**：day-1 玩家手里有什么
3. **资源稀缺曲线**：普通资源多易得？魔晶多稀有？决定游戏节奏
4. **图标词汇表清单**：基础物品类型 + 修饰符库（与 §9 相关）

### 中优先级（影响玩家体验）
5. **死亡掉落具体规则**：掉哪些、掉多少、随机还是按比例
6. **镇的离线 tick 速率**：1:1 / 1:24 / 暂停
7. **NPC 创造的频率与质量上限**：避免 NPC 把市场刷爆

### 低优先级（可以先不解，但要记着）
8. **镇的生命周期管理**：老镇怎么办、是否引入赛季
9. **法术书的写作机制**：玩家如何把脑内 API 落成书

## 12. MVP 范围建议（开发顺序）

1. **跑通单镇核心循环**（4-8 周）
   - 一个静态小镇（手工制作，不是 LLM 生成）
   - 5-10 个 NPC（基础 Smallville 实现）
   - 玩家可以采集、行动、对话
   - 基础 API ~10 个原语、mana 成本系统

2. **接入 LLM 创造物品**（2-4 周）
   - 玩家自然语言 → LLM 生成 Lua → 沙箱执行
   - 物品可以使用、可以销毁
   - 错误处理与友好反馈

3. **加入知识系统**（2-3 周）
   - 法术书作为可拾取物
   - NPC 教学
   - 逆向工程

4. **加入 NPC 创造与社会反应**（4-6 周）
   - NPC 模仿玩家创造
   - NPC 对玩家行为有记忆与反应
   - 经济流通

5. **战斗与死亡**（2-3 周）

6. **多镇支持**（4-6 周）
   - 镇生成 prompt 系统
   - 玩家镇分配
   - 多角色账号

> 总计 MVP 约 6-8 个月，前提是有 2-3 人全职、且不卡在 API 设计上反复推翻。
